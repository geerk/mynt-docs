The goal of this guide is to get you up and running quickly with mynt. If after reading this guide you still have questions that aren't answered by the other doc pages or you run into any issues, feel free to open an issue on GitHub or hop in IRC.

{# # INSTALLATION #}

Installation

The recommended way to install mynt is via pip. If you already have pip installed, the below should be all you need. Otherwise, head on over to the pip site to get it installed.

$ pip install mynt

After mynt has been installed, simply run the mynt command with the --version flag to confirm everything completed successfully.

$ mynt --version
mynt {{ site.version }}
{# # DIRECTORY STRUCTURE #}

Directory structure

Let's start off by initializing a new site. This is done with the init subcommand.

$ mynt init ~/projects/my-mynt-site/

Throughout the rest of this guide, the directory you initialized your site in will be referred to as the project root. Given the above usage of init, the project root would be ~/projects/my-mynt-site/.

If the directory you want to initialize your site in already exists, you will have to pass the -f flag to force initialization.

$ mynt init -f ~/projects/my-mynt-site/

In doing so, mynt will delete that directory in the process. For example, if the directory my-mynt-site already exists in ~/projects/, the above command will delete my-mynt-site before recreating it and initializing your site there.

Now that you have your site initialized, there are two general rules governing how directories and files are handled by mynt.

The first of which being any directory or file who's name begins with an underscore or a period will be ignored. The one exception to this rule being the three special directories that we will talk about shortly. However, do note that you can have ignored directories and files within these three special directories.

The second rule is that outside of the assets and posts directories (two of the three special directories), all files that aren't HTML or XML files are ignored.

This rule is handy if you want to have a README or LICENSE file at the root of your project. Since neither of those files are HTML or XML files, mynt will ignore them.

Now, onto those three special directories. In a mynt site there are three special directories: _assets/, _posts/, and _templates/. These three directories are all located at the project root.

The assets directory

The assets directory is where everything that isn't a template lives. CSS files, images, JavaScript libraries, and so on. If you want mynt to recognized a file that isn't an HTML or XML file, it has to live somewhere in the assets directory.

One place ignored directories can be useful is in the assets directory. For example, if you are using a CSS preprocessor like SASS, you can put your SASS files in _assets/css/_sass/, have SASS output to _assets/css/, and all of your SASS files will be ignored by mynt.

The posts directory

You likely guessed it. The posts directory is where all of your posts will live. There isn't really anything else to say about this directory.

Ignored directories can also be put to use in the posts directory. For example, if you have some post ideas you are working on, but aren't ready to publish them, you can put them in _posts/_drafts/ and mynt will ignore them.

The templates directory

Lastly, there's the templates directory. The name of this directory can be a bit confusing for the sole reason that any HTML or XML file that isn't in either the assets or posts directories is a template. They will all get passed through the configured renderer (Jinja by default).

The difference between having a template in the templates directory and somewhere else in your site, is that templates in the templates directory are not actual pages. They are simply templates that get reused by other templates or are used by mynt internally (like a post's layout).

It may help clear things up if you think of the templates directory as an includes directory.

{# # CONFIGURATION #}

Configuration

The configuration of mynt is handled by a single file at the root of your project: config.yml. If the file doesn't exist a set of defaults will be used, but if you wish to change any settings you'll have to create it. All of the settings are rather straight forward, so I won't go into detail here, but if that's what you're looking for, check out the config documentation.

One powerful thing about mynt's configuration file, is that everything in it, mynt setting or not, is made available to templates. This can be taken advantage of in many ways. For example, you could add a title or author attribute and then if you ever need to change that information, all you have to do is update it in your site's config rather than in every template.

{# # TEMPLATES #}

Templates

With your site set up and a general sense of how a mynt site is structured, let's move on to templates.

In the case of this guide (and also by default) the renderer used is Jinja. If you're not already familiar with Jinja, head on over and give the Jinja documentation a quick read.

For the most part a template in mynt is no different from any other Jinja template. However, there are three things worth mentioning. The first being two mynt specific helper functions and the second as well as the third being the available globals and locals.

Helper functions

Two mynt specific helper functions are provided to make retrieving URLs within your site easier. They do this by taking into account how your site is configured. Specifically the assets_url, base_url, and domain settings.

The first helper function is get_asset(). You should use it whenever you need a URL to anything living in the assets directory. The second helper function is get_url() and it should be used whenever you need a URL to anywhere else in your site.

Whenever you need a link internal to your site, you should use these two helper functions. The reason being, if you ever decide to move it, you'll only need to update the relevant configuration settings to update all of the links.

Globals

Template globals are variables containing site data that are available in every template. There are four template globals: site, posts, archives, and tags.

The first, site, is simply a dictionary of everything in the site's config file. The other three all contain your site's posts each organized in a different way. First there is posts, which is just a list. Then there is archives, which is a dictionary organized by year and month. Lastly there is tags, which is organized by tag and ordered by post count then name.

For a detailed look at the structure of the template globals, check out the globals section of the templates documentation.

Locals

Template locals are very similar to the globals we just discussed with the one exception that they are only available in specific templates. There are three template locals: post, archive, and tag.

Each local is a specific item of it's respective global that is made available to the template defined by it's respective layout setting.

For example, post is only available to templates that are used as a post's layout and contains the data of a single post. Similarly, archive is only available to archive_layout and contains the data of a single year of the archives global. And lastly, tag is only available to tag_layout and contains the data of a single tag.

For a detailed look at the structure of the template locals, check out the locals section of the templates documentation.

{# # POSTS #}

Posts

Now with templates out of the way, let's move on to posts.

The first thing you need to know about posts is that post filenames need to be formatted in a specific way.

YYYY-MM-DD[-HH-MM]-An-example-post.md

The hour and minute is optional and if not supplied the hour and minute of when you last edited the post will be used. In the case of having multiple posts on the same day, if you edited them in the order you want them displayed, you don't need to supply the hour and minute.

Now let's take a look at how posts need to be formatted themselves.

With the site you've initialized (and by default), the markup language used in posts is Markdown. There are only two things that differentiate a post from an ordinary Markdown document. The first being the YAML frontmatter and the second being the ability to use the features of the configured renderer.

YAML frontmatter

The YAML frontmatter is a block of YAML fenced by three dashes at the start of a post containing post metadata. It looks like the following:

---
layout: post.html
---

Every post requires a YAML frontmater with the layout attribute set. The layout attribute tells mynt which template in the templates directory it should use to render the post.

The only other attribute that has a special meaning in the YAML frontmatter, though it's optional, is tags. The tags attribute is used by the tagging functionality mynt offers which we'll talk about later.

You are free to add whatever data you want to a post's YAML frontmatter and it will be available in templates when accessing that specific post. A common example of this is using an attribute to store the formatted version of a post's title.

There are many ways to make use of the YAML frontmatter. For example, you can implement simple multiple author support by adding an author attribute to every post. Then a specific author's posts can be grabbed by filtering a for loop or using the groupby filter.

{{ "{% for post in posts if post.author == 'Author' %}
    ...
{% endfor %}" }}
{{ "{% for author, posts in posts|groupby('author') %}
    ...
{% endfor %}" }}

Renderer functionality

All of the features of the configured renderer are also available for use in posts with the exception of what data is available. The only template data available to posts is the YAML frontmatter of the post itself and the site template global.

---
layout: post.html
title: An example post.
---

# {{ '{{ title }}' }}

![image]({{ '{{ get_asset(\'images/image.jpg\') }}' }})

For example, in the above post, Jinja functionality is used to create a header containing the value of the title attribute from the YAML frontmatter as well as a template helper to grab a link to an image.

{# # TAGGING #}

Tagging

With templates and posts covered, let's move on to how mynt handles tagging.

To tag a post, simply add the tags attribute to the YAML frontmatter containing a list of the desired tags.

---
layout: post.html
title: An example post
tags: [Foo, Bar]
---

Tags are case-sensitive. "Foo" and "foo" are two different tags.

Once a post has been tagged, this change is reflected in a couple of ways. First, all of a post's tags will be available via a tags attribute. Second, that post will be added to each tag in the tags global.

Automatic tag pages

If your site has a lot of tagged posts it may be desirable to have a separate page for each tag. One of mynt's key features is the ability to do just that.

To enable it, tag_layout must be set in your site's config. This setting tells mynt which template in the templates directory to use when rendering these pages. Once it is set, a page will be rendered with that template for each tag.

{# # ARCHIVES #}

Archives

We've gone over posts and tagging so now let's talk about archives.

For the most part, how you want to handle post archives is up to you. To give you as much flexibility as possible there are the three template globals. Most notably, archives, which is a dictionary of your site's posts organized by year and month.

Automatic archive pages

Just like with tags, another key feature of mynt is the ability to automatically generate archive pages. If your site has a lot of posts or if you want to provide an alternate more compact way of browsing all of your site's posts, such a feature may be of use to you.

Enabling this feature is done very much in the same way as enabling automatic tag pages. In your site's config, set archive_layout to the template in the templates directory you want mynt to use when rendering these pages. Doing so, a page will be rendered with that template for each year a post was made.

{# # GENERATION #}

Generation

Now that we have the basics covered, it's time to generate your site.

Generating a mynt site is handled by two different subcommands: gen and watch. If you just need to make a quick change, gen is best for the job. It will generate your site and then exit.

$ mynt gen <source> <destination>

If you're going to be making a bunch of changes to your site, constantly reusing gen will quickly become a hassle. Instead you can use the watch subcommand.

$ mynt watch <source> <destination>

For the most part it is identical to gen. However, there is one major difference: Every time you make a change to your site, it will be automatically regenerated for you.

When using either subcommand, if the destination already exists, you will have to pass the -f flag to force generation.

$ mynt {gen,watch} -f <source> <destination>

In doing so, mynt will delete any directory or file at the root of the destination that doesn't begin with an underscore or period.

Now that you've generated your site, I'm sure you want to check it out and see the fruits of your efforts. If you have a local server setup you're good to go, but if you don't, mynt's got you covered. The last subcommand we're going to talk about is serve.

$ mynt serve <source>

This subcommand should not be used to publicly host your site. It should only be used for local development and testing.

To start up the server pass the directory containing your generated site to the serve subcommand. Then, to view your site simply browse to http://localhost:8080/.

For a more detailed look at the commands mynt provides, check out the command line documentation.

Once you're happy with your site, you're good to upload it to whichever host you wish, whether that be GitHub Pages or your own VPS.
